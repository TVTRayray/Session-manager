# 补充需求与重构文档 (PRD-3)

此文档针对此前版本出现的排版失效痛点发出了重构级的要求，详细定义了终端界面(TUI) 下的面板调整计算规则，并确立了对 UI 组件进行系统性自动化测试的机制。

## 1. 鼠标交互与选中态 (Mouse Focus & Highlight)
* **功能描述：** 当用户通过鼠标直接点击或使用快捷键选中面板（侧边列表栏 或 主内容栏）时，被激活的当前焦点 (Focused) 面板必须在视觉上进行明显的高亮强调（Highlight）。
* **UI 设定：** 可以通过诸如：选中的边框(Border)变色、背景加深、标题文字高亮加粗等终端样式来给出即时反馈，从而告诉用户后续的所有调整大小动作将会作用于哪一个面板。

## 2. 界面分栏与 Size 动态调整重构 (Layout & Re-sizing Refactor)

鉴于前序版本在此功能上的反复与不稳定，特对底层约束逻辑重新下发指令：

### 2.1 零和排版规则
* 整个试图空间被**严格锁定为两块区域**。即水平下的左、右，或垂直下的上、下。
* **联动机制：** 扩大当前被 Focused 的面板大小(Size)，就意味着严格同步缩减另一块副面板的 Size。

### 2.2 尺寸精密量化调整 (Dimension Sizing)
* **交互设定：** 使用 `Ctrl+Shift+=` (`+`号键) 作为扩大面板快捷键，使用 `Ctrl+Shift+-` (`-`号键) 作为缩小面板快捷键。
* **步长单位：** 注意在命令行 TUI 开发中不存在像素 (1px) 概念，最小单位是“字符块”。所以：
  * 当处于**水平排版 (Horizontal)**：是对面板的 Width 进行调整，步长建议设定为扩大/缩小 **5 列 (Columns)**。
  * 当处于**垂直排版 (Vertical)**：是对面板的 Height 进行调整，由于行距原因，步长建议设定为放大/缩小 **2 到 3 行 (Rows)**。
* **防卫性截断底线 (Min-Size Box)：** 任何面板都不能被缩减到 0。内部框架在处理计算指令时必须拦截边界，例如：最窄不能小于 15 列，最矮不能小于 5 行。如果触及底线则该减小操作静默不处理。

### 2.3 布局强切与重绘生命周期 (Layout Switch Rendering)
* **准确定义：** 用户触发 `Ctrl+Shift+H`（水平）与 `Ctrl+Shift+V`（垂直）不属于表层的样式变动，而是整个终端布局约束树结构的重建。
* **渲染要求：** TUI 程序接收到此宏指令后，必须完全丢弃现有的横向/纵向 Layout Constraint，采用全新反转的方向去重新挂载上述两块面板，并手动发起一次全屏重绘强制刷新事件 (Force Screen Redraw/Re-render)，以防出现画面拖影或更新滞后（这就是该功能此前经常失效的通病）。

## 4. 质量保障：集成 `test-tui` 进行终端自动化测试

之所以之前的布局和尺寸调整多次返工，就在于终端层缺乏自动化模拟与兜底。
* **策略引入：** 在本次重构中，**强制引入系统内置的 `test-tui` Skill**（可通过查看本地 `.agent/skills/test-tui` 了解使用规范）作为我们的 UI 调试器和自动化验证器。
* **具体执行与验收思路：**
  * 在开发/重构过程中，必须添加 `RUST_LOG="trace"` 或指定 `log_dir` 并使用 `just codex` 指令启动。
  * 需要建立交互式的 TUI 测试脚本：通过程序化的方式先后发送 `Text Input` 和 `Enter` ，并向 TUI Emit 给定的虚拟快捷配置按键（模拟 `Ctrl+Shift+=` ）。
  * 框架应该能在 debug log 里明确打印出“面板大小设定已变更为 X 行/列”并验证屏幕组件刷新过关。这一步需验证通过后才能确认重构的 Size Resize API 稳定可靠。
