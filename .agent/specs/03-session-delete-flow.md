# Spec 03 - Session Delete Flow

## 基本信息

- Spec ID：`03-session-delete-flow`
- 标题：安全删除与选中态恢复
- 当前状态：`qa-passed`
- 关联阶段：`阶段二：Manage Sessions`
- 当前责任角色：`QA`

## 背景与目标

- 浏览能力完成后，产品的第二条核心链路是删除无用会话，以释放本地存储并保持列表整洁。
- 删除属于高风险文件系统操作，必须独立为一个 vertical slice，避免与阅读体验实现交叉污染，同时确保 QA 可以单独验证安全边界。
- 本 slice 在 Rust 中实现文件删除和状态恢复，需要把路径校验、确认状态和错误返回建模为显式类型，而非隐式字符串约定。

## 用户故事 / 成功标准

- 作为：需要清理本地历史会话的用户
- 我希望：在左侧选中某个会话后，通过快捷键安全删除该文件，并立即看到列表与详情同步刷新
- 从而：不离开 TUI 就能完成管理动作，同时避免误删和状态错乱

成功标准：

- [x] 左侧选中会话后，按 `d` 或 `Delete` 进入删除确认流程。
- [x] 删除前必须出现二次确认弹窗，默认焦点为取消或否定选项。
- [x] 仅当用户明确确认后才执行文件系统删除。
- [x] 删除成功后，目标会话从左侧列表移除，右侧清空或切换到邻近项，不出现索引越界。
- [x] 删除失败时，列表和右侧详情保持原状，并在状态栏中展示失败原因。

## 非目标

- 不支持批量删除。
- 不支持撤销删除、回收站或软删除。
- 不扩展为文件重命名、移动或编辑功能。

## 接口与契约

- 输入：
  - 当前选中 `SessionListItem.abs_path`
  - 用户确认结果：`confirm` 或 `cancel`
- 输出：
  - 删除成功后的新 `session_list`
  - 删除失败时的结构化错误消息
- IPC / API：
  - 删除动作必须通过受限文件操作接口执行，接口内部校验目标路径归属。
  - UI 层不得直接拼接路径并调用裸删除。
- 异常 / 错误返回：
  - 目标路径不在 `~/.codex/sessions` 下时直接拒绝，并记录安全错误。
  - 文件不存在、权限不足、IO 失败时返回可展示错误，不得导致程序退出。

## 数据与状态变化

- 新增状态：
  - `delete_modal_state`
  - `pending_delete_target`
  - `delete_result_message`
- 变更状态：
  - 进入确认弹窗时冻结列表主操作，仅响应确认/取消。
  - 删除成功后重算 `selected_index` 和右侧详情。
- 持久化影响：
  - 永久删除目标 `.jsonl` 文件。

## 边界与失败场景

- 删除首项：成功后选中新的首项或空态。
- 删除末项：成功后选中前一项，若无前一项则进入空态。
- 删除时目标已被外部移除：视为失败并刷新状态，不得崩溃。
- 连续快速触发删除键：在确认框存在期间不得重入第二个删除流程。

## 实施要求

- 删除默认焦点必须偏向安全，不允许“回车即确认删除”。
- 路径校验必须基于规范化后的绝对路径，防止符号链接或路径拼接绕过目录限制。
- 删除后选中态恢复策略必须在实现前固定：优先选择原位置的新项目，否则退到前一项，否则空态。
- 状态栏消息必须区分“取消删除”和“删除失败”，避免误导用户。
- Rust 文件系统操作必须返回结构化错误，并在 UI 层完成用户可见文案映射，不允许直接把底层 panic 暴露到终端。
- 路径安全校验优先使用 Rust 标准库的规范化和前缀判断能力实现，避免手写字符串裁剪逻辑。
